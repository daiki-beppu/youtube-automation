# subagent 委譲オーケストレーション規約

- **Status**: Accepted
- **対象**: 実作業を subagent へ委譲する skill
- **関連規約**: [Sonnet-safe スキル記述規約](skill-authoring-guidelines.md)

## Hard Gates

subagent 委譲を行う skill は、次の条件をすべて満たさなければならない。

1. `workflow-state.json` の読み書きはメインエージェントだけが行う。
2. `AskUserQuestion` による承認ゲートはメインエージェントだけが通過させる。承認が必要な操作は、承認されるまで subagent へ委譲しない。
3. subagent は実作業ツールを実行するが、`workflow-state.json` へ書き込まない。
4. メインエージェントは subagent の完了報告だけを信頼せず、期待する成果物のファイル存在と、現在の `phase` に対する整合を検証してから state を更新する。

## 責務境界

### メインエージェント専任の責務

メインエージェントはオーケストレーターとして、次を専任する。

- `workflow-state.json` を読み、現在の `phase` と既存成果物から次に委譲する作業を決める。
- 外部反映または破壊的操作の前に、対象・件数・取消可否などを提示し、`AskUserQuestion` の明示的な実行 / 中止の 2 択で承認を得る。中止または無回答なら委譲せず停止する。
- subagent の終了後、委譲時に列挙した期待成果物がすべてファイルとして存在することを確認する。
- 成果物が完了後の `phase` の前提を満たすことを確認し、整合する場合だけ `workflow-state.json` の成果物情報、`phase`、`updated_at` を更新する。

成果物が存在しない、または `phase` と整合しない場合は state を更新しない。欠落・不整合を具体的に報告し、同じ委譲を再実行できる状態で停止する。

### subagent の責務と禁止事項

subagent は委譲された一つの実作業に集中し、次を行う。

- 指定された入力パスを読み、委譲プロンプトで指定された実作業ツールを実行する。
- 期待成果物を指定パスへ生成する。
- 完了報告形式に従い、実行結果と生成した成果物パスをメインエージェントへ返す。

subagent は次を行ってはならない。

- `workflow-state.json` への書き込み、`phase` の変更、成果物情報や `updated_at` の更新。
- `AskUserQuestion` による承認取得、または承認が必要な外部反映・破壊的操作の先行実行。
- 指定されていない入力の探索、成果物の追加、後続 phase の作業。

## 委譲フロー

1. メインエージェントが `workflow-state.json` を読み、現在の `phase` を確定する。
2. 前提ファイルの存在と妥当性を確認する。満たさない場合は前工程を案内して停止する。
3. 承認ゲートが必要なら、メインエージェントが `AskUserQuestion` で承認を得る。承認前は委譲しない。
4. 下記テンプレートを埋め、一つの完了条件に絞って subagent へ委譲する。
5. subagent の完了後、メインエージェントが期待成果物のファイル存在を確認する。
6. 成果物と完了後の `phase` の整合を確認する。
7. すべて PASS の場合だけ、メインエージェントが `workflow-state.json` を更新する。一つでも FAIL なら更新せず停止する。

## 委譲プロンプトのテンプレート

角括弧内を具体値で置き換える。入力と成果物は、subagent が追加判断しなくてよいようにリポジトリルート相対パスで列挙する。

```text
あなたは [作業名] の実作業担当です。以下の範囲だけを実行してください。

## 入力パス
- [入力ファイルまたはディレクトリのパス]
- [追加の入力パス]

## 実行する作業
- [使用する skill / CLI / script と具体的な引数]

## 期待成果物
- [成果物パス]: [機械的に確認できる完了条件]
- [追加の成果物パス]: [機械的に確認できる完了条件]

## 禁止事項
- workflow-state.json を読み書きしない
- AskUserQuestion を実行しない
- 指定範囲外の作業や後続 phase を実行しない

## 完了報告形式
status: success | failure
artifacts:
  - path: [生成した成果物パス]
    result: [完了条件を満たしたことを示す値]
errors:
  - [failure の場合のエラー内容。success の場合は空配列]
```

メインエージェントは `status: success` を state 更新の根拠にしない。`artifacts` に報告された各パスを実際に確認し、期待成果物との完全一致と `phase` 整合を検証する。

## Sonnet-safe 7 ルールとの整合

本規約は [Sonnet-safe スキル記述規約](skill-authoring-guidelines.md) の7ルールに追加して適用する。

| ルール | subagent 委譲での適用 |
|---|---|
| ① 発動キーワードの相互排他・否定トリガー | オーケストレーターと実作業 skill の発動条件を相互排他にし、進捗確認だけの場合などの否定トリガーも `description` に書く。 |
| ② 外部反映 / 破壊的操作の承認ゲート | メインエージェントが `AskUserQuestion` の明示2択で承認を得るまで、該当作業を subagent へ委譲しない。 |
| ③ 前提ファイルの存在ガード | 委譲前にメインエージェントが入力パスを確認し、欠落時は前工程を案内して停止する。 |
| ④ 判断基準なき判断要求の禁止 | 入力、実行内容、成果物、完了条件をテンプレートへ具体的に記載し、subagent に「適切に」などの判断を委ねない。 |
| ⑤ 同一ロジックの散文重複禁止 | 検証・生成ロジックは `references/` の単一ソースへ置き、委譲プロンプトには呼び出しと引数だけを書く。 |
| ⑥ 完了条件・Hard Gates は冒頭に | 本規約の責務境界を skill の冒頭60行以内の Hard Gates / 完了条件にも記載する。 |
| ⑦ 解決できない参照の地の文配置禁止 | subagent が解決できる入力パスだけを渡し、未接続・アクセス不能な参照は実行手順に含めない。 |

## 実装時チェックリスト

- [ ] メインエージェントだけが `workflow-state.json` を読み書きする。
- [ ] 承認ゲートをメインエージェントが担当し、承認前の委譲を禁止している。
- [ ] subagent の実作業ツールと、生成する成果物パスが具体的である。
- [ ] 委譲プロンプトに入力パス、期待成果物、完了報告形式がある。
- [ ] メインエージェントによるファイル存在と `phase` 整合の検証がある。
- [ ] 検証 PASS 後にだけ state を更新し、FAIL 時は state を更新せず停止する。
