## 前後工程

- `前工程`: `なし`
- `後工程`: `/benchmark`, `/discover-competitors`

## Hard Gates

- このスキルは **状態を持たない読み取り専用の調査**である。`config/channel/analytics.json::benchmark.channels` を含む config schema、既存 TTP、調査入力を変更しない。TTP の自動入替は行わない。
- 既定の成果物は会話内レポートだけ。ユーザーがこの実行について明示的に「保存して」と依頼した場合だけ `docs/research/market-<YYYY-MM-DD>.md` を生成する。依頼がなければディレクトリもファイルも作らない。
- 保存先に同日ファイルがすでに存在する場合は、上書き前に既存ファイルが置換されることを示し、「上書きする / 会話内だけにする」の明示 2 択で確認する。承認されるまで書き込まない。
- 根拠は URL またはローカルパス、確認日、対応する主張を必ず記録する。`references/report-contract.md` の採用閾値を満たさない候補は「根拠不足」とし、TTP 入替候補や有望ニッチとして推奨しない。
- 調査結果から `/discover-competitors`、`/benchmark`、config 更新を自動実行しない。次アクションとして提案するだけに留める。

## 完了条件

`references/report-contract.md` の 7 セクションを会話内に提示し、各候補を `候補` / `保留（根拠不足）` / `非推奨` のいずれかに分類し、保存依頼の有無に応じた分岐を完了した時点で終了する。保存ありの場合は同じ内容が `docs/research/market-<YYYY-MM-DD>.md` に存在することも確認する。

## Overview

既存チャンネル運用から任意に呼び出し、現在の TTP と外部候補、または複数のニッチ仮説を同じ評価軸で比較する。結果は意思決定材料であり、設定変更や TTP 入替そのものではない。

用途の違い:

| skill | 主目的 | 既定成果物 |
|---|---|---|
| `/market-research` | 調査問いに対する横断比較、TTP 入替候補・ニッチ仮説の提示 | 会話内レポート |
| `/discover-competitors` | YouTube Data API による未登録チャンネルの検索・スコアリング | ranking Markdown + CSV |
| `/channel-new` 分析モード | 収集済み benchmark / comments の詳細分析 | `docs/channel-research.md` ほか |

## Instructions

### Step 1: 調査問いと比較条件を固定する

次の 4 項目を会話から抽出する。欠けている項目だけをユーザーへ短く確認し、揃うまで調査を始めない。

1. 調査問い: 1 文で答えられる問い
2. 比較対象: 現行 TTP、候補チャンネル、またはニッチ仮説を合計 2 件以上
3. 評価軸: 3〜5 軸。例は需要シグナル、競合密度、更新継続性、自チャンネルとの再現可能性、差別化余地
4. 対象市場と観測期間: 言語 / 地域、および「直近 90 日」などの期間

既存チャンネルの `config/channel/analytics.json::benchmark.channels` や `docs/channel/` 以下の TTP メモが存在する場合は読み取り専用の比較入力にできる。存在しなくても、ユーザーが比較対象を指定できるため停止条件にはしない。

### Step 2: 根拠を収集する

- 現在値を扱うため、Web 検索または接続済みの一次情報を使って確認する。ユーザーが Web 検索を禁止した場合はローカル成果物だけを使い、その制約を「不確実性」に記録する。
- チャンネル / 動画の直接 URL、公式ページ、YouTube 上の公開指標、既存 benchmark 成果物を優先する。検索結果の要約だけを根拠にしない。
- 根拠ごとに `E1`, `E2`, ... の ID、URL またはローカルパス、確認日、観測事実、支える主張を記録する。
- 数値は比較対象と観測期間を揃える。揃えられない数値は直接比較せず、「不確実性」に差を記録する。
- 根拠が見つからない主張を一般知識で補完しない。

### Step 3: 同じ評価軸で比較する

比較対象 × 評価軸の表を作り、各セルを `強い` / `中立` / `弱い` / `判定不能` と根拠 ID で埋める。根拠のないセルは必ず `判定不能` とする。

TTP 入替候補は、現行 TTP より優れると観測できた軸と劣る軸を両方示す。「候補」はあくまで追加検証対象であり、既存 TTP の削除・置換を意味しない。

ニッチ仮説は「対象視聴者 / 視聴シーン / 満たす欲求 / 競合との差 / 最小の検証方法」の 5 要素で記述する。

### Step 4: 採用閾値を適用する

分類、根拠数、根拠不足時の扱いは `references/report-contract.md` をそのまま適用する。

- 閾値を満たす → `候補`
- 反証が根拠を上回る、または自チャンネルで再現不能 → `非推奨`
- それ以外、または根拠不足 → `保留（根拠不足）`

### Step 5: 会話内レポートを返す

`references/report-contract.md` の見出し順で、調査問い、比較対象、根拠、TTP 入替候補、ニッチ仮説、不確実性、次の検証を提示する。候補には必ず根拠 ID と分類を添える。

### Step 6: 明示依頼がある場合だけ保存する

保存依頼がない場合は「会話内のみ・ファイル未生成」と伝えて終了する。

保存依頼がある場合だけ、現在の日付を `YYYY-MM-DD` で取得し、会話内レポートと同じ内容を `docs/research/market-<YYYY-MM-DD>.md` に保存する。親ディレクトリがなければこの時だけ作成する。保存後はパスを提示する。既存ファイルがある場合は Hard Gates の上書き確認を先に行う。

## 障害時ガイダンス

| 状況 | 対処 |
|---|---|
| 比較対象が 1 件以下 | 追加対象をユーザーに確認し、2 件以上になるまで停止する |
| Web / API に接続できない | ローカル根拠だけで続け、満たせない候補は `保留（根拠不足）` にする |
| 指標の観測期間が揃わない | 直接比較せず `判定不能` とし、再取得条件を次の検証に書く |
| 保存先が既にある | 上書きの明示承認がなければ書き込まず、会話内レポートだけを残す |
