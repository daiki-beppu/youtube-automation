# 変更の分類と影響度判定ルール

`/release-notes` スキル Phase 2 / Phase 3 で参照する判定ルール集。

---

## Phase 2: 変更カテゴリ判定

各 PR / コミットを 🆕 / 🔧 / 🐛 / 🧹 に分類するルール。

### 判定フローチャート

```
PR メッセージの type を見る
├─ feat:
│   ├─ 新規ファイル (新 skill / 新 CLI / 新モジュール) → 🆕 新機能
│   └─ 既存ファイルの挙動拡張 → 🔧 挙動変更
├─ fix: → 🐛 バグ修正
├─ refactor:
│   ├─ 運営者の挙動に影響 → 🔧 挙動変更
│   └─ 内部リファクタのみ → 🧹 内部改善
├─ docs: / test: / chore: / ci: / perf: → 🧹 内部改善
└─ 不明 → PR 本文の "Closes #N" の issue を読む
```

### 判定の補助質問

迷ったら以下を順番に問う:

1. **運営者が新しいコマンドを覚える必要があるか？** → YES なら 🆕 新機能
2. **同じコマンドを実行したときの挙動が変わるか？** → YES なら 🔧 挙動変更
3. **これまで失敗していた操作が成功するようになるか？** → YES なら 🐛 バグ修正
4. **上記いずれも NO** → 🧹 内部改善

### 例（v5.5.0 から）

| PR | 種別 | カテゴリ | 判定根拠 |
|---|---|---|---|
| #245 (playlist スキル新規) | feat | 🆕 | 新 skill ファイル追加、運営者が `/playlist` という新コマンドを使えるように |
| #244 (metadata-audit スキル新規) | feat | 🆕 | 同上 |
| #324 (masterup Step 6 rsync 化) | fix | 🔧 | `fix` だが既存挙動を変える（取り残されていた素材が同期されるように） |
| #325 (wf-next main repo 検出) | feat | 🔧 | `feat` だが既存スキルの拡張、同じ `/wf-next` の挙動が変わる |
| #326 (collection-ideate freshness 統一) | docs | 🔧 | `docs` だがドキュメントの記述変更が運営者の挙動理解に影響、軽微なら 🧹 でも可 |
| #248 (channel-setup 設定 push モード) | refactor | 🔧 | frontmatter description 更新で発動条件が変わる、挙動変更 |
| #230 (yt-channel-settings 400 fix) | fix | 🐛 | 純粋なバグ修正、これまで失敗していた push が成功するように |
| #246 (wf-new に populate-scene-phrases 統合) | feat | 🆕 / 🔧 両義 | 多言語チャンネルは新機能、単一言語チャンネルは影響なし。両方で言及 |

### 「両義」になる変更の扱い

1 つの PR が複数カテゴリにまたがる場合（例: 機能追加 + 既存挙動変更）、**主要影響のカテゴリ** に分類し、副次影響は本文で言及。

## Phase 2: 重大変更 Top 5 判定

TL;DR テーブルと「重大変更」の Top 5 を抽出する判定基準。優先度高い順に:

1. **下流必須対応を伴う変更**: 下流側で操作不要では完了しない（例: tag bump、local fix 衝突）
2. **日常操作に直接影響する変更**: ユーザーが毎日触る skill / CLI の挙動変更
3. **これまで失敗していた操作が成功するようになる変更**: 致命的なバグ修正
4. **新機能で利用可能になる主要 skill / CLI**
5. **設定スキーマ変更や認可スコープ変更などの破壊的変更**

5 件以下しか該当しない場合は無理に埋めない（Top 3 / Top 4 でも OK）。

## Phase 3: ダウンストリーム影響度マトリクス

各下流リポジトリ × 各変更を以下に分類:

| 影響度 | 記号 | 意味 | 例 |
|---|---|---|---|
| **必須対応** | 🔴 | 操作しないと壊れる / 動かない | tag bump、`uv lock` 更新、`yt-skills sync` 実行 |
| **確認推奨** | 🟡 | 動作するはずだが念のため検証推奨 | 設定 schema 互換性確認、認証フローの再認証 |
| **影響なし** | 🟢 | 操作不要 | 内部リファクタのみ、対象外の機能 |

### 判定の優先順位

1. `pyproject.toml` の **tag pin 形式** → tag bump 必須 → 🔴
2. `pyproject.toml` の **main 追従形式** → `uv lock` で自動取り込み、ただし lockfile 更新が必要 → 🔴
3. **local fix の検出** → 衝突解消必須 → 🔴
4. **新 skill / CLI**:
   - 配布されるが使うかどうかは下流次第 → 🟢（強制ではない）
   - ただし `yt-skills sync` 自体は実行必要 → 🔴（既に上記で含まれる）
5. **挙動変更**: 下流の運用フローを変える可能性 → 🟡
6. **バグ修正**: これまで失敗していた操作の運営者は確認推奨 → 🟡
7. **内部改善**: → 🟢

### マトリクス例

| 変更 | rjn (tag pin) | deepfocus365 (main 追従) |
|---|---|---|
| tag bump 必要 | 🔴 必須 | 🟢 不要 |
| uv lock 更新 | 🔴 必須 | 🔴 必須 |
| yt-skills sync 実行 | 🔴 必須 | 🔴 必須 |
| local fix 衝突解消 | 🟡 確認推奨 | 🔴 必須（既知の local fix あり） |
| 新 skill /playlist 利用 | 🟢 任意 | 🟢 任意 |
| yt-channel-settings 400 fix の恩恵 | 🟡 確認推奨 | 🟡 確認推奨 |
| 内部処理高速化 | 🟢 影響なし | 🟢 影響なし |

このマトリクスはアップグレードガイドの「あなたのチャンネルへの影響」セクション（プレーンテキスト箇条書き）の根拠にする。出力時はテーブルではなく `- <リポジトリ>: <現状> → <やること>` 形式に整形する。

## 内部改善カテゴリ（🧹）の書き方

運営者影響なしを明示しつつ、参考程度に箇条書きで 3〜5 件。例:

```markdown
### 🧹 内部改善（運営者影響なし、参考のみ）

技術的なリファクタやテスト追加が多数行われましたが、運営者の操作には影響しません。代表的なもの:

- 内部処理の高速化（戦略分析が 7.7 倍高速に）
- コスト計算のハードコード単価を撤廃し GCP Cloud Console を一元ソースに
- OAuth ハンドラのリファクタ、ストリーミング系のヘルスチェック整備

詳細は `CHANGELOG.md` の v<ver> セクションを参照してください。
```

詳細を書き始めるとキリがないので、**代表的な 3〜5 件で打ち切る** のがコツ。「詳細は CHANGELOG.md」で逃げる。
