# 提出前セルフ監査チェックリスト

実装完了を宣言する前に、以下 8 項目を根拠（ファイル:行）付きで照合する。本リポジトリの過去レビュー指摘 371 件の頻出パターンから抽出した基準であり、1 項目でも未照合のまま完了を宣言してはならない。

## 監査 8 項目（サマリー）

| # | 項目 | 照合内容 |
|---|------|----------|
| 1 | 受入条件の充足表 | issue の要件を 1 項目ずつ引用し、実装行とテストを紐付ける。未充足があれば完了宣言しない |
| 2 | 挙動差分 ↔ テスト 1:1 | 追加・変更した観測可能な挙動それぞれに対応する assert が存在する |
| 3 | 実経路テスト | テストが公開入口から production コードを通る。「production 側のどの行を消すと落ちるか」に答えられる |
| 4 | 兄弟入口・配線の貫通 | 契約変更は同責務の全入口に適用。設定値は定義→読込→実行→出力まで貫通 |
| 5 | ドキュメント突き合わせ | 変更した契約を SKILL.md / docstring / examples / CHANGELOG と照合し旧文言を残さない |
| 6 | 失敗を成功に見せない | catch・fail-soft が失敗を exit code / Result / status に反映している |
| 7 | 既存契約の退行禁止 | base 側の公開契約・既存テストの削除には廃止根拠がある |
| 8 | スコープ検査 | diff が issue の対象範囲のみ。ついで改善・無関係差分がない |

## 各項目の詳細

### 1. 受入条件の充足表（頻出度: 28 件）

issue の受入条件を 1 項目ずつ引用し、「対応する実装（ファイル:行）」「対応するテスト」を表にして出力する。1 つでも未充足なら `Closes #N` を外し部分対応と明記する。「設定可能に」という要求はハードコードでは充足しない（default 値と override 値の両方が効くこと）。

### 2. 挙動差分 ↔ テスト 1:1（頻出度: 72 件・最多）

diff で追加・変更した観測可能な挙動（ログ文言、exit code、retry、fallback、エラー分岐、新しい状態遷移）を列挙し、各項目に対応する assert が新規・既存テストに存在することを 1 対 1 で照合する。バグ修正は「修正前なら落ちるテスト」を必ず 1 本含める。

### 3. 実経路テスト（頻出度: 41 件）

fake / mock / 手注入 / monkeypatch が検証対象の配線そのものを迂回していないか点検する。CLI は実引数解釈から、HTTP は実 endpoint から検証する。テスト内に production ロジックのコピーを書かない。部分文字列の存在確認だけで「挙動を検証した」としない。

### 4. 兄弟入口・配線の貫通（頻出度: 29+10 件）

契約（データ形式、config キー、タグ処理等）を変更したら、同じデータ・同じ責務を扱う入口を grep で全列挙し（別 CLI、別 provider、Python/TS 両実装、preflight と実行系）、全入口に適用するか、適用しない理由を成果物に明記する。設定キー・CLI option を追加したら実際に値を変えて JSON → loader → 実行経路 → 出力まで貫通させる。定義だけして使わないフィールドを残さない。

### 5. ドキュメント突き合わせ（頻出度: 36 件）

変更した契約・手順・数値（件数、パス、回数）を grep で全ドキュメント（SKILL.md frontmatter と本文、config コメント、docstring、examples、CHANGELOG）に突き合わせる。旧文言の残存と、新規記載の事実誤り（実装は 10 件なのに「9 件」と書く等）の両方を検査する。

### 6. 失敗を成功に見せない（頻出度: 23 件）

catch・fail-soft・警告化を書くたびに「この失敗を握った結果、呼び出し側とユーザーは何を誤認するか」を確認する。永続化失敗・部分完了は必ず exit code / Result / HTTP status に反映する。生 `Exception` の catch は禁止（プロジェクト規約）。捕捉した例外の詳細はログに残す。

### 7. 既存契約の退行禁止（頻出度: 15 件）

置換・リファクタでは base 側の公開契約（overload、エラー文言、許容入力、標準ライブラリ由来の挙動）と既存テストを列挙し、新実装への移行先を 1 件ずつ確認する。既存テストの削除は「契約自体の廃止根拠」を issue から引用できる場合のみ。

### 8. スコープ検査（頻出度: 20 件）

`git diff --stat` を issue の対象範囲と照合し、無関係ファイルを revert する。要件外の「ついで改善」は成果物に follow-up 候補として記録し、diff には含めない。既存挙動・既存フローの削除には issue 上の根拠を引用する。

## 機械チェック（提出前に必ず実行）

lint / format / test / 未使用コード検出をローカルで一括実行し、すべて green を確認してから完了を宣言する。新規追加行に `typing.Any` / `: any`、What/How を言い換えるだけの説明コメント、フラグ引数（bool で挙動を切り替える関数）がないか grep する（いずれもレビューポリシー明文の REJECT 条件）。
